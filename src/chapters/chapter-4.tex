\section{Tujuan Eksperimen}

Eksperimen pada penelitian ini bertujuan untuk menjawab tiga hal.
Pertama, apakah dinamika sinkronisasi CQRS dapat dimodelkan sebagai sistem linier dalam kerangka state-space.
Kedua, apakah kontroler LQR yang dirancang dari model tersebut mampu mengatur parameter sinkronisasi secara adaptif.
Ketiga, bagaimana performa LQR dibandingkan dengan pendekatan lain, yaitu parameter statik, aturan berbasis ambang batas, dan PID.

\section{Lingkungan Eksperimen}

Seluruh eksperimen dijalankan pada lingkungan yang sama untuk menjaga konsistensi hasil.
Arsitektur sistem uji mengikuti rancangan pada Bab III (lihat Gambar~\ref{fig:arsitektur-desain}), terdiri dari write model, message broker, message sink, dan read database.

Komponen read database menggunakan Elasticsearch versi \esversion{} dengan alokasi \escpus{} CPU, memori \esmemory{}, dan Java heap \esjavaheap{}.
Konfigurasi ini dijaga tetap selama seluruh eksperimen.

\section{Eksperimen Open-Loop dan Identifikasi Sistem}

Sebelum merancang kontroler, perlu diketahui terlebih dahulu bagaimana sistem merespons perubahan parameter.
Untuk itu dilakukan eksperimen open-loop, di mana parameter sinkronisasi divariasikan tanpa kontroler aktif dan respons sistem dicatat.
Data dari eksperimen ini kemudian digunakan untuk mengestimasi model state-space.

\subsection{Prosedur Eksperimen}

Eksperimen open-loop dilakukan pada tanggal \openloopdate{}.
Sistem sinkronisasi dijalankan tanpa kontroler aktif, dan parameter batch size serta poll interval divariasikan secara sistematis.
Variabel state dicatat setiap \openloopinterval{} detik, menghasilkan total \openloopsamples{} data point.

Variabel state yang diukur adalah jumlah antrian pesan (\emph{queue length}), utilisasi CPU, utilisasi memori, operasi I/O tulis, dan rata-rata latensi.
Variabel kontrol yang divariasikan adalah batch size dan poll interval.

\subsection{Hasil Identifikasi Sistem}

Dari data tersebut, model state-space $x_{k+1} = Ax_k + Bu_k$ diestimasi menggunakan regresi linier.
Matriks A berukuran 5$\times$5 dan matriks B berukuran 5$\times$2.

Kualitas model diukur menggunakan koefisien determinasi ($R^2$) untuk setiap variabel state (Tabel~\ref{tab:sysid-fit}).

$R^2$ yang rendah pada memori dan I/O kemungkinan disebabkan oleh faktor non-linear seperti garbage collection dan caching yang sulit ditangkap model linier.

Meski demikian, variabel utama yang menjadi fokus pengendalian, yaitu queue length dan latensi, memiliki model yang cukup akurat untuk digunakan sebagai dasar perancangan kontroler LQR.

\section{Perancangan Kontroler}

Berdasarkan model yang diperoleh, beberapa strategi pengendalian dirancang dan diuji.
Selain LQR, tiga baseline dan empat varian ANN juga digunakan sebagai pembanding.

\subsection{Konfigurasi LQR}

Sebagaimana dibahas pada Bab II, matriks Q pada LQR menentukan seberapa besar penalti yang diberikan ketika masing-masing variabel state menyimpang dari kondisi yang diinginkan.
Matriks Q berbentuk diagonal, di mana setiap elemen diagonal berkorespondensi dengan satu variabel state sesuai urutan vektor state $\mathbf{x} = [queue\_length, cpu, mem, io, latency]^T$.
Semakin besar nilai elemen diagonal untuk suatu variabel, semakin keras sistem berusaha menekan deviasi variabel tersebut, bahkan jika itu berarti variabel lain harus dikorbankan.

Empat varian Q dirancang untuk menguji skenario prioritas yang berbeda (Tabel~\ref{tab:lqr-variants}).
Matriks R dijaga sama untuk seluruh varian agar perbedaan perilaku berasal dari Q saja.

\begin{table}[H]
    \centering
    \caption{Varian pembobotan LQR}
    \label{tab:lqr-variants}
    \renewcommand{\arraystretch}{1.4}
    \begin{tabular}{|c|l|l|}
    \hline
    \textbf{Varian} & \textbf{Prioritas} & \textbf{Diagonal Q} \\
    \hline
    Q1 & Prioritas backlog & diag(10, 1, 1, 1, 1) \\
    \hline
    Q2 & Prioritas utilisasi sumber daya & diag(1, 10, 10, 10, 1) \\
    \hline
    Q3 & Prioritas latensi & diag(1, 1, 1, 1, 10) \\
    \hline
    Q4 & Seimbang & diag(1, 1, 1, 1, 1) \\
    \hline
    \end{tabular}
\end{table}

Pada Q1, elemen pertama (queue length) diberi bobot 10 sedangkan sisanya 1.
Artinya, deviasi backlog dianggap 10 kali lebih ``mahal'' dibanding deviasi CPU, memori, I/O, atau latensi.
Kontroler akan berusaha menurunkan backlog lebih dulu, meskipun utilisasi sumber daya menjadi kurang efisien.

Pada Q2, bobot 10 diberikan pada tiga variabel sumber daya sekaligus (CPU, memori, I/O).
Kontroler akan menjaga agar utilisasi sumber daya tetap rendah, meskipun backlog bertambah.
Varian ini merepresentasikan skenario di mana kapasitas server terbatas dan overload harus dihindari.

Pada Q3, bobot 10 diberikan pada latensi saja.
Kontroler akan memprioritaskan penurunan waktu tunggu sinkronisasi, yang secara tidak langsung juga menekan backlog karena pemrosesan dipercepat.

Pada Q4, seluruh elemen diagonal bernilai 1 sehingga tidak ada variabel yang diprioritaskan.
Varian ini menjadi baseline untuk melihat perilaku LQR tanpa preferensi tertentu.

\subsection{Konfigurasi Baseline}

Tiga strategi baseline digunakan sebagai pembanding.

\begin{enumerate}
\item \textbf{Statik.} Batch size tetap \staticbatch{} dan poll interval tetap \staticpoll{} ms. Tidak ada penyesuaian parameter.

\item \textbf{Rule-based.} Parameter disesuaikan berdasarkan dua aturan sederhana.
Jika queue length melebihi 5000, batch size dinaikkan 50.
Jika utilisasi CPU melebihi 80\%, batch size diturunkan 50.
Poll interval dijaga tetap.

\item \textbf{PID.} Kontroler PID mengatur batch size berdasarkan selisih queue length terhadap target nol.
Parameter yang digunakan adalah $K_p = 0{,}1$, $K_i = 0{,}01$, $K_d = 0{,}05$ dengan poll interval tetap pada 1000 ms.
PID hanya mengontrol satu variabel (batch size) berdasarkan satu sinyal error (queue length).
\end{enumerate}

\subsection{Konfigurasi ANN}

Empat varian kontroler berbasis Artificial Neural Network (ANN) juga diuji.
Masing-masing varian (ann\_q1 sampai ann\_q4) dilatih dengan fungsi biaya yang sesuai dengan varian Q pada LQR, sehingga perbandingan antara LQR dan ANN dilakukan secara adil.

\section{Analisis Hasil}

Bagian ini membahas temuan utama dari eksperimen closed-loop, mencakup perbandingan antar kontroler dan pengaruh pembobotan Q terhadap perilaku sistem.

\subsection{Perbandingan LQR terhadap Parameter Statik}

Dari keempat skenario, LQR-Q3 dengan prioritas latensi adalah satu-satunya varian yang konsisten menghasilkan backlog lebih rendah dari parameter statik.
Pada ramp, Q3 mencapai backlog 9355 dibandingkan statik 10324 (penurunan 9,4\%).
Pada step, Q3 menghasilkan 10883 vs statik 14157 (penurunan 23,1\%).

Namun, keunggulan ini diperoleh dengan biaya utilisasi CPU yang lebih tinggi.
Pada step, CPU Q3 mencapai 16,0\% dibanding statik 7,1\%.
Hal ini sesuai dengan perilaku Q3 yang memperkecil poll interval (rata-rata 297 ms vs 1000 ms statik), sehingga sistem lebih sering melakukan polling dan pemrosesan.

Varian lain (Q1, Q2, Q4) umumnya tidak menunjukkan keunggulan signifikan terhadap statik dalam hal backlog, bahkan beberapa menghasilkan backlog lebih tinggi.
Ini menunjukkan bahwa pemilihan bobot Q sangat berpengaruh terhadap efektivitas LQR.

\subsection{Perbandingan LQR terhadap PID}

PID secara konsisten menghasilkan performa terbaik di semua skenario dari sisi backlog dan throughput.
Backlog PID pada step hanya 6403 (vs LQR-Q3 10883), dan throughput PID 700,6 msg/s (vs LQR-Q3 132,4 msg/s).

Penyebab utamanya adalah PID menggunakan batch size yang jauh lebih besar (rata-rata 550--751) dibanding LQR (120--196).
Batch besar memungkinkan pemrosesan lebih banyak pesan per siklus, sehingga backlog turun lebih cepat.

PID hanya mengontrol batch size berdasarkan satu sinyal, yaitu queue length, dengan poll interval tetap 1000 ms.
LQR mengontrol batch size dan poll interval sekaligus berdasarkan seluruh state.
Pada konfigurasi saat ini, pengendalian LQR yang lebih menyeluruh ini belum menghasilkan performa yang melampaui PID.

Salah satu kemungkinan penyebabnya adalah rentang batch size yang digunakan LQR terlalu kecil.
Batch size rata-rata LQR berkisar 120--196, jauh di bawah PID yang mencapai 750.
Perbedaan ini terlihat pada Gambar~\ref{fig:timeseries-batch}. PID (kuning) menaikkan batch size hingga mendekati batas atas, sedangkan LQR (hijau) bergerak dalam rentang yang lebih sempit.
Penyesuaian rentang operasi dan kalibrasi ulang matriks Q dan R berpotensi meningkatkan performa LQR.

\subsection{Pengaruh Pembobotan Q}

Variasi bobot Q menghasilkan perilaku yang berbeda dan konsisten di semua skenario.

\begin{enumerate}
\item \textbf{Q1 (backlog priority)} menghasilkan backlog dan perilaku yang tidak jauh berbeda dari statik.
Bobot 10 pada queue length tampaknya belum cukup besar untuk menghasilkan perubahan signifikan pada aksi kontrol.

\item \textbf{Q2 (resource priority)} menghasilkan backlog tertinggi di semua skenario, namun dengan CPU terendah (3,2--4,9\%).
Sesuai prioritasnya, Q2 menghemat sumber daya dengan mengorbankan kecepatan sinkronisasi.
Poll interval Q2 juga paling panjang (rata-rata 1737--1915 ms), artinya sistem jarang melakukan polling.

\item \textbf{Q3 (latency priority)} konsisten menghasilkan backlog terendah di antara varian LQR.
Poll interval-nya paling pendek (293--380 ms) dan utilisasi CPU paling tinggi (12--18,8\%).
Varian ini paling agresif dalam mengurangi latensi.

\item \textbf{Q4 (balanced)} berada di antara Q1 dan Q2, tanpa keunggulan khusus.
Pada periodic step, Q4 menunjukkan anomali berupa throughput yang sangat rendah.
\end{enumerate}

\subsection{Keterbatasan dan Temuan}

Selama eksperimen, ditemukan beberapa keterbatasan yang perlu diperhatikan untuk pengembangan selanjutnya.

\begin{enumerate}
\item Model linier memiliki $R^2$ rendah untuk utilisasi memori (\rsqMem) dan operasi I/O (\rsqIo).
Kedua variabel ini kurang terprediksi oleh model, yang menjadi keterbatasan pendekatan LQR.

\item Rentang aksi kontrol LQR (batch size 120--196) jauh lebih kecil dari PID (550--751).
Hal ini membatasi kemampuan LQR dalam mengurangi backlog secara agresif.
Perlu dilakukan kalibrasi ulang agar LQR dapat menggunakan rentang batch size yang lebih lebar.

\item Pada skenario periodic step, LQR-Q4 dan seluruh varian ANN menunjukkan anomali berupa throughput dan utilisasi nol.
Kemungkinan penyebabnya adalah kontroler menghasilkan parameter di luar rentang operasional yang valid, atau terjadi timeout pada eksekusi.
Hasil ini perlu diinvestigasi dan eksperimen perlu diulang.

\item Eksperimen dilakukan dengan satu kali pengulangan per kombinasi.
Hasil belum dapat diklaim signifikan secara statistik tanpa repetisi yang memadai.

\end{enumerate}
